## Motivation

There are use cases that reasonably demand users to prove ownership over their World ID using a specific blinding commitment. At the time of writing there are two well known use cases: authentication with issuers, and authentication to generate `session_id_r_seed`s.

### Issuer Authentication

Some examples of use cases which warrant issuers to request authentication from users:
- **Re-issuance**. Allowing the user to get a new copy of their credential due to loss.
- **Status information**. Providing the user with information on their current enrollment state.
- **Cancellation/deletion/unregistration**. Allowing the user to unregister, delete or cancel their existing credential.

For the cases above, issuers need to know the legitimate owner of a Credential is requesting the operation. Issuers do not know the user's `leafIndex` by design, so a mechanism to authenticate based on the Credential's subject (`sub`) must be devised.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

This specification introduces a new circuit to the Protocol: **"Ownership Proof"**. The Ownership Proof is a new circuit which allows users to prove _ownership_ (as defined below) using a specific querying input.

At a high level, the circuit MUST perform the following validations:

1. Prove knowledge of the secret key (`sk`) matching -> public key (`pk`)
2. Prove membership of `pk` in the list of authenticator public keys `pk[]`
3. Prove commitment of `pk[]` for authenticator public keys
4. Prove commitment of `pk[]` matches the leaf index in the Merkle Tree
5. Prove `H(domainSeparator, leafIndex, blindingFactor)`
