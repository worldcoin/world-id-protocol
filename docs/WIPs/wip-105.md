## Abstract

This spec defines a standard protocol for World ID authenticators to communicate with each other through a relay bridge. The primary use case is enabling a new authenticator to request insertion into an existing World ID account by pairing with an already-registered authenticator. Communication is established through a deeplink or QR code, relayed through an untrusted bridge service, and secured with end-to-end encryption and a user-verified pairing code.

## Motivation

World ID 4.0 introduces multi-authenticator accounts: a single World ID can have up to 7 registered authenticator keys in the `WorldIDRegistry`. Adding a new authenticator requires an existing authenticator to sign an [`InsertAuthenticator`](https://docs.rs/world-id-primitives/latest/world_id_primitives/api_types/struct.InsertAuthenticatorRequest.html) transaction that includes the new authenticator's EdDSA public key and on-chain address.

This creates a bootstrapping problem. The new authenticator must securely transmit its public keys to an existing authenticator, which may be running on a different device, platform, or network. Without a standard communication protocol:

1. **Interoperability breaks.** Each authenticator provider would define its own pairing mechanism, preventing cross-provider authenticator addition. A user with Authenticator A could not add Authenticator B from a different provider without both providers agreeing on a proprietary protocol.
2. **Cross-device pairing has no path.** The most common scenario — adding a web-based authenticator on a laptop from a mobile authenticator — requires communication between two devices that may not share a local network.
3. **Security is inconsistent.** Without a defined verification mechanism, implementations may omit user confirmation entirely, exposing users to man-in-the-middle attacks during pairing.

This spec defines a universal inter-authenticator communication protocol that any conforming authenticator can use to pair with any other, regardless of provider, platform, or device.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

The terms [Authenticator](https://docs.rs/world-id-core/latest/world_id_core/struct.Authenticator.html) and [AuthenticatorPublicKeySet](https://docs.rs/world-id-primitives/latest/world_id_primitives/authenticator/struct.AuthenticatorPublicKeySet.html) are as defined in the World ID Protocol.

### Definitions

- **Requesting Authenticator**: A new authenticator that wants to be added to an existing World ID account.
- **Approving Authenticator**: An existing, registered authenticator that authorizes the addition.
- **Bridge**: An untrusted relay service that forwards encrypted messages between two authenticators without being able to read them. The reference implementation is the [wallet-bridge](https://github.com/worldcoin/wallet-bridge).

### Protocol Overview

```mermaid
sequenceDiagram
    participant r as Requesting Auth (new)
    participant b as Bridge
    participant a as Approving Auth (existing)

    r->>r: Generate session ID + X25519 keypair
    r->>b: Connect(session_id)
    r->>r: Display QR code / deeplink

    a->>a: Scan QR / open deeplink
    a->>a: Generate X25519 keypair
    a->>b: Connect(session_id)
    a->>r: Ephemeral X25519 public key (plaintext)
    Note over r,a: Encrypted channel established (X25519 + ChaCha20-Poly1305)

    r->>a: Insert request (pk, addr, label)
    a->>a: Compute verification code from pk
    r->>r: Display verification code
    Note over r,a: User enters code shown on Requesting into Approving
    a->>a: Verify code, prompt user to approve
    a->>r: Confirmation (status: ok)
    a->>a: Sign & submit InsertAuthenticator
    r-->>a: Ack + optional push URL
```

The protocol has four phases:

1. **Connection** — The Requesting Authenticator generates a session, connects to the bridge, and displays a connection URI (as a QR code or deeplink). The Approving Authenticator opens this URI and connects to the same bridge session.
2. **Key Exchange** — Both parties perform X25519 key agreement to establish an end-to-end encrypted channel over the untrusted bridge.
3. **Request & Verification** — The Requesting Authenticator sends its public keys through the encrypted channel. The user verifies the pairing by entering a code displayed on the Requesting Authenticator into the Approving Authenticator.
4. **Confirmation** — The Approving Authenticator acknowledges the request and may proceed with the on-chain `InsertAuthenticator` transaction.

### Connection URI

The Requesting Authenticator initiates a session by generating and displaying a connection URI:

```
worldid://bridge/v1/e?i={session_id}&pk={ephemeral_pk}&b={bridge_url}
```

| Parameter | Required | Description |
|-----------|----------|-------------|
| `i` | REQUIRED | Unique session identifier, generated from a CSPRNG. Minimum 16 bytes, hex-encoded. |
| `pk` | REQUIRED | The Requesting Authenticator's ephemeral X25519 public key, hex-encoded (32 bytes). Used for key agreement, not to be confused with the authenticator's EdDSA key. |
| `b` | OPTIONAL | The bridge service base URL. If omitted, implementations MUST use a default bridge URL defined in their configuration. |

This URI MUST be presentable as a QR code (for cross-device pairing) or as a clickable deeplink (for same-device scenarios). The `worldid://` URI scheme MUST be registered by authenticator implementations that support acting as an Approving Authenticator.

### Bridge Session & Encryption

1. The Requesting Authenticator MUST generate a fresh X25519 keypair for each session and connect to the bridge using the session ID.
2. Upon opening the connection URI, the Approving Authenticator MUST generate its own ephemeral X25519 keypair and connect to the same bridge session.
3. The Approving Authenticator MUST send its ephemeral X25519 public key to the Requesting Authenticator as the first (unencrypted) message on the bridge.
4. Both parties MUST derive a shared secret using X25519 Diffie-Hellman and encrypt all subsequent messages with ChaCha20-Poly1305 ([RFC 8439](https://www.rfc-editor.org/rfc/rfc8439)).
5. ChaCha20-Poly1305 nonces MUST be sequential per sender, starting from zero, incremented for each message.
6. The bridge MUST NOT be assumed to provide confidentiality or integrity. All security properties derive from the end-to-end encryption between the two authenticators.

### Message Types

Messages sent through the encrypted bridge channel are encoded as URIs. Future specs MAY define additional message types under the `worldid://auth/v1/` namespace.

#### Insert Request

After the encrypted channel is established, the Requesting Authenticator sends:

```
worldid://auth/v1/insert?pk={offchain_pk}&addr={onchain_addr}&label={label}
```

| Parameter | Required | Description |
|-----------|----------|-------------|
| `pk` | REQUIRED | The Requesting Authenticator's EdDSA public key on the BabyJubJub curve (compressed, hex-encoded, 32 bytes). This key will be added to the account's [`AuthenticatorPublicKeySet`](https://docs.rs/world-id-primitives/latest/world_id_primitives/authenticator/struct.AuthenticatorPublicKeySet.html). |
| `addr` | OPTIONAL | The Requesting Authenticator's on-chain Ethereum address (checksummed hex, [ERC-55](https://eips.ethereum.org/EIPS/eip-55)). This address is required for the on-chain `InsertAuthenticator` transaction. If omitted, it MUST be provided in a subsequent message before insertion can proceed. |
| `label` | OPTIONAL | A human-readable label for the new authenticator (e.g., `"Chrome on MacBook"`). MUST NOT exceed 64 UTF-8 bytes. |

The Approving Authenticator MUST reject the insert request if the account's `AuthenticatorPublicKeySet` already contains 7 keys (the maximum defined by `MAX_AUTHENTICATOR_KEYS`).

### Verification Code

To prevent man-in-the-middle attacks on the bridge relay and to confirm user intent, a verification code MUST be presented before the Approving Authenticator proceeds.

**Computation.** The verification code is derived deterministically from the Requesting Authenticator's EdDSA public key:

1. Let `pk_bytes` be the compressed EdDSA public key (32 bytes, as transmitted in the `pk` parameter of the insert request).
2. Compute `h = SHA-256(0x57494431 || pk_bytes)`, where `0x57494431` is the ASCII encoding of `"WID1"` (domain separator).
3. Interpret the first 4 bytes of `h` as a big-endian `uint32`.
4. Compute `code = uint32 mod 1_000_000`.
5. Format as a zero-padded 6-digit string (e.g., `"042817"`).

**Requesting Authenticator** — MUST compute the verification code from its own EdDSA public key and display it prominently to the user with instructions to enter it on the Approving Authenticator.

**Approving Authenticator** — MUST:
1. Display the new authenticator's details (including `label` if provided, and a truncated fingerprint of the `pk`).
2. Prompt the user to enter the 6-digit verification code shown on the Requesting Authenticator.
3. Verify the entered code matches the code computed from the received `pk`.
4. Only proceed if the code is correct **and** the user explicitly approves the insertion.

A failed verification code entry SHOULD allow the user to retry. After 3 consecutive failures, the Approving Authenticator SHOULD terminate the session.

### Confirmation

After successful verification, the Approving Authenticator sends through the bridge:

```
worldid://auth/v1/status?result={result}
```

| Value | Description |
|-------|-------------|
| `ok` | The Approving Authenticator will proceed with the on-chain insertion. |
| `rejected` | The user denied the request. |
| `error` | A technical error prevented processing (e.g., account already has 7 authenticators). |

On receiving `result=ok`, the Requesting Authenticator MAY respond with an acknowledgment:

```
worldid://auth/v1/ack?push={push_url}
```

| Parameter | Required | Description |
|-----------|----------|-------------|
| `push` | OPTIONAL | An HTTPS URL that accepts POST requests for future notifications (e.g., when the on-chain transaction is confirmed). |

The push URL, if provided, MUST use HTTPS. The Approving Authenticator MAY use it to notify the Requesting Authenticator of the transaction status but is not required to.

### Session Lifecycle

1. Either party MAY close the bridge connection at any time. Closing before confirmation is sent MUST be treated as an implicit rejection by the receiving party.
2. Bridge sessions SHOULD expire after a maximum of 5 minutes. Implementations MUST NOT keep idle sessions open indefinitely.
3. All ephemeral key material (X25519 private keys, shared secrets) MUST be securely erased when the session ends.

## Rationale

**Relay bridge over direct peer-to-peer.** Direct communication (Bluetooth, local network discovery, WebRTC) was considered but the primary use case — cross-device pairing — involves authenticators on different networks (e.g., mobile data vs. corporate WiFi). A relay avoids NAT traversal complexity and works across all network configurations. End-to-end encryption ensures privacy even if the bridge is compromised or malicious.

**QR codes and deeplinks.** The `worldid://` URI scheme integrates with OS-level app registration on both mobile (Universal Links, App Links) and desktop. QR codes naturally encode URIs and are the established mechanism for cross-device pairing in protocols like WalletConnect and passkey registration. Deeplinks handle same-device scenarios seamlessly.

**User-entered verification code.** A 6-digit numeric code provides ~20 bits of entropy, sufficient for interactive verification where an attacker has a single real-time opportunity. Code entry (rather than visual comparison) was chosen because it requires deliberate action on the Approving Authenticator, reducing accidental approvals. The `"WID1"` domain separator in the hash input prevents cross-protocol collisions if the same key is used elsewhere.

**URI-encoded bridge messages.** Messages use the `worldid://` URI scheme for consistency with the connection deeplink. URIs are human-readable (aiding debugging) and extensible via new query parameters without breaking existing parsers. The `worldid://auth/v1/` namespace allows future message types (e.g., for authenticator removal or credential exchange) to coexist.

**Ephemeral keys and forward secrecy.** Each bridge session uses fresh X25519 keypairs discarded after the session. This provides forward secrecy: compromising a future session key cannot reveal past pairing data. Ephemeral keys also prevent the bridge `pk` in the connection URI from being used to track users across pairing attempts.
