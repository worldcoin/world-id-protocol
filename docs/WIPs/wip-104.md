## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.ietf.org/rfc/rfc2119.txt).

To the ends described above, a distinction is introduced between two classes of Authenticators:
1. **Proving Authenticator**: A "Proving Authenticator" is capable of authenticating a user for Proof generation and presentation. A Proving Authenticator SHALL NOT perform any management operations on the World ID, i.e. it cannot perform any state changes on the `WorldIDRegistry` for the user (e.g. inserting an authenticator, removing an authenticator).
2. **Admin Authenticator**: An "Admin Authenticator" is an Authenticator capable of all operations for a user's World ID, including all state changes on the `WorldIDRegistry` as well as generating and presenting proofs.
  - **Transitory**. Prior to the introduction of this spec, all registered authenticators MUST be considered Admin Authenticators.

Going forward, any "management operations" on a World ID can be understood as any operation which results in a state change over a user's World ID in the `WorldIDRegistry`. _Examples of this are: inserting an authenticator or updating a Recovery Agent._

### `WorldIDRegistry` Updates
1. The `WorldIDRegistry` (or "registry" in this spec) MUST be updated to allow inserting, removing and updating authenticators without a corresponding management key (_A management key is represented as an `address` in the `_authenticatorAddressToPackedAccountData` variable_).
